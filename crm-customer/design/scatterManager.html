散客管理--手动散客新增saveScatterCustomer序列图设计
RULE-A1前台页面校验：
1)客户名称、联系人姓名、联系方式为必填项，输入客户名称后，联系人姓名默认为客户名称，可以更改但不能为空，手机号码、固定电话至少录入一个；
2)手机号码为11位，超出11位不显示，固定电话文本框内提示：如：021-31350606-12；
3)商机状态不可编辑，散客类型默认为临代散客，不可编辑
4)身份证号码、税务登记证号需要校验有效性，校验规则参照固定客户的身份证、税号校验规则
5)下拉框无默认值，选择后可以清空
6)营业部、经理新增散客时，所属部门默认为本部门，且不可修改；客服小组人员、开单组成员新增散客时，所属部门为所辖区域内的营业部； 新增散客时，所属部门不能为空
7)客户账号信息可以为空,若有值时，其中的信息都必填,每个散客的账号信息不重复
8)临欠额度：正整数，小于等于3000元
后台校验-校验是否重复：
1)系统根据“手机号码”判断固定客户是否存在相同信息，若存在，不允许保存，否则提示：联系方式XXXX与XXX营业部的固定客户编码XXXX重复，不能保存为散客
2)根据“部门+手机号码”判断本部门散客信息是否存在相同信息，若存在，不允许保存，否则提示：联系方式XXXX与散客客户编码XXXX重复
3)若手机号码不存在，则通过“联系人姓名+固定电话”判断固定客户信息是否存在相同信息，若存在，则不允许保存，提示：联系方式XXXX与XXX营业部的固定客户编码XXXX重复，不能保存为散客
4)若手机号码不存在，根据“部门+联系人姓名+固定电话”判断本部门散客是否存在相同信息，若存在，不允许保存，否则提示：联系方式XXXX与散客客户编码XXXX重复
5)审批中散客、审批中固定客户的手机号码或者姓名+固定电话，若新增的散客和审批中散客、审批中固定客户的手机号码或者姓名+固定电话一致，则不允许创建
6)不同的散客可以有相同的银行账号，同一个散客下的银行账号不能重复
1)散客客户编码规则规定：大写字母S+创建日期+8位随机码（例：S20120921-25896374）
2)散客类型:临代散客、普通散客
3)账户性质:“外部单位账户”“外部个人账户”
4)手动新增散客，散客类型默认为“临代散客”
散客管理--系统自动生成散客无
RULE-A11）同步到CRM系统后，生成一条客户编码，客户编码规则为：大写字母S+创建日期+8位随机码（例：S20120921-25896374）
2）散客类型默认为普通散客；
散客管理--散客修改updateScatterCustomer序列图设计
RULE-A1散客修改权限：
1. 散客归属部门操作人员以及客服小组可修改散客信息；
2. 否则提示“只有散客归属部门人员或者客服人员有修改权限！”
RULE-A21. 修改客户名称、联系人姓名、联系方式时，系统校验规则参考散客新增业务规则”
RULE-A3 1. 除了所属部门、客户编码、商机状态不可修改，其余信息都可修改；
2. 修改/新增账号信息，只能由部门收银员进行操作，同时，CRM系统生成一条工作流，由部门经理审批；
3. 散客的账号信息修改后，该条账号信息还在审批中，则此账号信息被锁定，不支持业务操作；
4. 审批中的散客信息不允许修改；
5. 散客信息审批中，系统显示修改前的信息，审批同意后，信息覆盖原有的信息，审批退回后，原信息保持不变；
6. 账号信息删除为逻辑删除；
7. 普通散客增加临欠额度或者账号信息后，状态发生变化，由普通散客转变为临代散客，散客信息可同步到FOSS，该状态不可逆；
8. 普通散客信息修改除临欠额度和账号信息外的字段，状态不发生变化，不同步到FOSS；
散客管理--散客查询queryScatter序列图设计
RULE-A1散客查询权限要求：
1. 经理及以下人员只能查询归属于本部门的散客信息，归属部门默认为当前用户所在部门；
2. 区域经理可以查询本小区营业部的散客信息；
3. 大区总经理可以查询本大区营业部的散客信息；
4. 事业部总裁可以查询本事业部营业部的散客信息；
5. 客户管理组可以查询全国营业部的散客信息；
6. 客服小组可以查看/维护所辖营业部的散客信息；
7. 开单组可以查看所辖营业部的散客信息
RULE-A2查询条件要求：
1. 客户名称、联系人姓名模糊查询，其余为精准查询；
2. 查询条件所属部门不能为空；
3. 查询优先级：客户编码＞手机号码＞固定电话＞其他，当输入客户编码、手机号码、固定电话时，查询结果不受创建时间限制，其他查询条件都受创建时间限制；
4. 创建时间系统默认为三个月，从当前日期往前推三个月，最大可查询范围为一年；
5. 所属部门默认为当前用户所在部门；
RULE-A3查询结果：
1. 查询列表显示字段：客户状态、客户编码、客户名称、联系人姓名、手机号码、固定电话、临欠额度、合作意向、商机状态、货量潜力、行业、创建人、创建时间、最后修改人、最后修改时间；
2. 选择一条散客，双击或者点击查看详情按钮，进入散客详细信息界面，里面的内容只可查看，不可修改。
3. 系统默认为20条，分页显示查询结果，按照创建时间的倒序排列，通过单击显示结果列标题实现以该列表头对当前页结果按顺序/逆序显示；
4. 能同步到FOSS的散客即为临代散客，在列表中用粉色区分，同时在列表下方有个温馨提示：粉色表示临代散客，开单时可开临时欠款和代收货款；
5. 已转化为固定客户的散客用橙色表示，同时在列表下方有个温馨提示：橙色表示该散客已经转化为固定客户。此类散客可以查看，不可删除，收银员可以修改账号信息；已转化为固定客户的散客，在列表中显示在普通散客后面；
散客管理--散客删除delScatter序列图设计
RULE-A11. 散客删除即逻辑删除，删除散客信息后，散客信息失效，包括：基本信息、业务信息、交易信息失效，该散客的联系方式可以再次使用
2. 散客删除后，不影响ERP/FOSS等的结算业务；
3. 审批中的散客信息不可删除，提示：该散客在审批中，工作流号xxxxxxx，审批未完成不能删除散客信息！
4. 散客删除或升级为固定客户后，在散客查询界面，只能通过散客客户编码查询出该条散客信息，点击修改按钮，账号信息可以新增、修改、删除，其他信息呈现灰色不可修改，账号信息修改权限为收银员，修改提交后，生成工作流，由部门经理审批；
散客管理--转成临代散客transferToFoss序列图设计
RULE-A11. 在散客查询界面选中普通散客，点击“转为临代散客”按钮，弹出提示框：是否确认将普通散客转化为临欠散客，选择是，普通散客转变为临代散客；
2. 在散客查询界面选中临代散客，点击“转为临代散客”按钮，弹出提示框：该散客已经为临代散客，不需要做转化；
3. 已同步到FOSS的临代散客，在CRM系统修改该条信息后，修改后的信息也需同步到FOSS，同步方式为实时同步；
散客管理--散客升级无
RULE-A11. 散客升级为固定客户时，如果归属部门有该条散客，且该散客有账号信息的，提示:该散客有X条账号信息，是否将账号添加到固定客户中？选择是，弹出账号信息选择界面；
2. 在账号信息选择界面，选中需添加的账号信息，点击确定，散客的账号信息添加到固定客户下，点击取消，账号信息添加失败；
3. 账号信息选择界面左下角有警示：请认真检查同步的账号信息是否正确！以免造成退错款！
4. 账号信息列表展示字段：开户行、开户姓名、银行账号、开户省市、支行名称、账户用途、财务联系人、手机号码、固定电话、是否默认账号、账户性质、账号与客户关系；
